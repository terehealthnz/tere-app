## Meta note on this exercise

This is a **solo tabletop** — one participant, no live external validation. Solo tabletops are less valuable than group tabletops because they miss the "wait, that's not how I understood it" moments between participants. This exercise should be repeated as a group tabletop once a technical second is appointed. The scenario chosen (AWS credential exposure) is deliberately close to a real event from 2026-07-07 — a rotated but shared IAM key. That makes the answers more grounded but also means some responses reflect very recent lived experience rather than latent process. Both are noted in the debrief.

## Timeline walkthrough

### T+0 — the incident begins (09:47 weekday)

**Scenario:** Contractor commits `.env.local` containing production `AWS_SECRET_ACCESS_KEY` to the private repo, cross-posts commit link to public Discord.

**Discussion questions and Tere's answers:**

**Q1: What detection controls would catch this?**
- git pre-commit hook running `git-secrets` or `detect-secrets` — **Tere does not currently have this.**
- GitHub secret scanning — **enabled by default for private repos, but scans a limited set of well-known credential patterns; AWS keys are covered.**
- Manual review before push — **not enforced.**

**Q2: Which controls does Tere currently have in place?**
- GitHub secret scanning (implicit, GitHub default). Will alert the repo admin (Patrick) via email if AWS key pattern is detected.
- No git-secrets pre-commit hook.
- No contractor onboarding document covering secrets handling.
- `.gitignore` covers `.env.local` but not `.env.local.backup` or other variants.

**Q3: What is the fastest realistic detection path today?**
- **GitHub secret scanning email** to the repo admin. Turnaround is usually minutes for well-known patterns like AWS keys.
- **AWS CloudTrail unusual activity** — only if configured with CloudWatch alarms. **Tere does not currently have CloudWatch alarms configured on the tere-bedrock IAM user's activity.**
- **User-reported error** — patients hitting rate limits caused by the attacker's traffic (as happened in the scripted scenario).

**Gap identified:** No proactive alarm on unusual IAM activity. Detection relies on GitHub secret scanning (which works but only for the specific commit path) or on downstream errors surfacing to patients.

### T+43 — detection

**Scenario:** Patient reports triage error. Investigation reveals Bedrock rate-limits and unknown CloudTrail invocations.

**Discussion questions and Tere's answers:**

**Q4: Immediate classification?**
- **P1 — High.** Credential compromise with active use, but no confirmed PHI exposure (the observed invocations are `hello world` tests, not customer data queries). Classify up if any evidence of data queries appears.

**Q5: Who is notified first, and how?**
- Compliance owner (Patrick) — who in this scenario is the person doing the investigation. **This is the single-person weakness noted in the DR plan.** If Patrick were unavailable, there is currently no documented handoff.

**Q6: First containment action, and does it carry any risk?**
- Deactivate the leaked IAM key in AWS IAM console (Tere already has direct AWS console access; Patrick is authenticated).
- Risk of the containment action itself: **any in-flight consultation would lose AI-note generation if the new key isn't in place before the next `/api/generate-notes` call**. But `ai_notes_enabled` flag can be flipped OFF as a fallback if the rotation window drags. Fax + LiveKit + Supabase all continue to work — Bedrock is the only affected sub-processor.
- **Practically:** in the actual 2026-07-07 rotation this was ~2 minute window; realistic.

### T+50 — containment complete

**Scenario:** Leaked key deactivated. New pair generated. Vercel env vars rotated. Redeploy triggered. Total containment: 7 minutes.

**Discussion questions and Tere's answers:**

**Q7: Was 7 minutes acceptable?**
- Yes for this scenario. The exposure window was already 43 minutes; another 7 for containment brings total to 50. Real damage window is bounded.
- Contrast: if the attacker had been exfiltrating data at high rate, 7 minutes could still be a significant volume. For Bedrock specifically the API is per-call — no bulk export vector.

**Q8: What if Patrick were unreachable for 3 hours?**
- **Current state: no formal handoff.** Nobody else has:
  - AWS console access
  - Vercel dashboard access
  - Supabase dashboard access
  - GitHub write access
- **This is a critical single-person weakness.** In a real 3-hour outage, damage compounds.
- Documented in DR plan Section 9 as a known weakness. Committed action: appoint a technical second before first PHO contract.

**Q9: How do you verify containment worked?**
- Hit `/api/bedrock-test` — should return `ok:true` with new-key-generated tokens.
- Check AWS CloudTrail for any subsequent invocations of the deactivated key (should return zero after deactivation).
- Check the Bedrock invocation logs by IP if available — attacker's IP should stop appearing.
- **Tere already has `/api/bedrock-test` verified working (built 2026-07-07).**

### T+70 — scope assessment

**Scenario:** CloudTrail shows 143 exploratory invocations in the exposure window. All appear to be small `hello world` queries. No customer data was involved.

**Discussion questions and Tere's answers:**

**Q10: Is this a notifiable breach under the Privacy Act 2020?**
- **Judgment call — leaning "no."** Reasoning: no PHI was accessed. The Bedrock API is stateless per-request; the attacker's invocations did not touch Tere's Supabase database, patient records, transcripts, or notes. The credential could have been used to access PHI if the attacker had crafted specific queries with Tere-owned prompts, but the evidence is they did not.
- **However:** if the exposure window had been longer, or if the invocation patterns showed queries with Tere-shaped prompts (suggesting the attacker had also compromised the codebase), the answer flips to "yes, notifiable."
- **Would seek external legal review before finalising this determination in a real event.** Legal counsel is committed action per the IR plan.

**Q11: Even if not notifiable, what internal actions should be taken?**
- Full incident report filed regardless of notifiability.
- Root cause analysis (see T+90).
- All secrets rotated as a precaution — not just AWS. Anything the attacker could have also seen if they had the same access path (Supabase service role, Stripe key, Twilio, LiveKit, Sentry). This is more work but the right defensive posture.
- Costs review — AWS may have billed for the 143 test invocations. Small (probably <$1) but tracked.

**Q12: Should any patient / provider / sub-processor be told?**
- **Patients:** No — the incident did not involve their data.
- **Providers:** Yes, informational — "we had a security event, contained, no PHI affected, here's what we're doing to prevent recurrence." Reduces surprise if they see any transient errors in the audit log.
- **Sub-processor:** AWS Support is notified as a courtesy and to request a review of the invocation logs to confirm no data queries occurred. This creates a paper trail from AWS's side confirming Tere's assessment.

### T+90 — root cause and durable change

**Scenario:** Root causes: no pre-commit hook, no contractor onboarding for secrets, `.gitignore` gaps, Discord amplification.

**Discussion questions and Tere's answers:**

**Q13: Which root cause is most durable to fix? Which is quickest?**

| Root cause | Time to fix | Durability |
|---|---|---|
| No pre-commit hook | ~1 hour (one-time) + adopting `git-secrets` or `detect-secrets` in dev instructions | **Very durable** — catches the class of incident, not just this one |
| No contractor onboarding on secrets | ~2 hours (draft document) + adopt as onboarding requirement | Durable if actually enforced |
| `.gitignore` gaps | ~5 min (add `.env*` catch-all pattern) | Durable, cheap |
| No policy on public discussion of work | ~1 hour (draft contractor code of conduct) | Fragile (relies on individual compliance) |

Most durable: **pre-commit hook.** Catches humans forgetting.

Quickest: **.gitignore fix.** Ship immediately.

**Q14: Draft the durable change list.**

Committed for this week:
1. Update `.gitignore` to catch `.env*` and `*.env.*` broadly.
2. Add `detect-secrets` or `git-secrets` pre-commit hook, documented in `docs/dev-setup.md` (or equivalent).
3. Draft `docs/contractor-onboarding.md` covering secrets handling, gitignore expectations, and public-discussion boundaries.
4. Enable CloudWatch alarm on tere-bedrock IAM user for unusual invocation volume.

Committed for next month:
5. Rehearse a full end-to-end rotation of every secret (not just AWS) so we know the timing and hit any documentation gaps.
6. Appoint a technical second with AWS + Vercel + Supabase + GitHub access.

**Q15: Who is accountable? By when?**

At current single-person scale: all actions are Patrick's. Once technical second appointed, split accordingly.

- Actions 1, 4 → this week

## Debrief summary

### What went well

- **Bedrock migration is fresh** — the runbook for AWS credential rotation was rehearsed on 2026-07-07 (one day before this exercise). Knowledge of the rotation process is current.
- **Existing feature flag system** (`ai_notes_enabled`) provides a real fallback if credential rotation drags — an option many organisations don't have.
- **`/api/bedrock-test` is already built and works** — verification of containment is a single URL hit.
- **Section 9 of the security-compliance doc already flagged single-person weakness** — this exercise confirms rather than surfaces that gap.
- **The IR plan classification tiers (P0-P3) worked cleanly** — the scenario naturally sat at P1, and the discussion identified conditions that would trigger upgrade to P0.
- **The response timeline (43 min exposure + 7 min containment) is within acceptable bounds** for the incident class.

### Gaps identified

1. **No pre-commit secret detection hook.** Detection relied on GitHub's built-in secret scanning, which works but is one control deep.
2. **No CloudWatch alarms on IAM user activity.** Anomalous invocations of the tere-bedrock user would go unnoticed unless downstream errors surfaced.
3. **Single-person weakness confirmed.** No handoff exists for AWS / Vercel / Supabase / GitHub console access. If Patrick is unavailable during an incident, containment stops.
4. **No formal incident report template.** IR plan Section 7 has a detection log template but not a full post-incident report format.
5. **`.gitignore` too narrow.** Only catches specific `.env.local`, not variants.
6. **No contractor onboarding document.** Contractor secrets handling relies on assumed knowledge.
7. **No documented policy on public discussion of work.** The Discord amplification in the scenario is realistic and covered by no current process.
8. **Legal counsel not on retainer.** Notifiability determination for edge cases (like this scenario) would benefit from external legal review; Tere has none engaged yet.
9. **Secrets-rotation dry-run has never been rehearsed for all sub-processors.** Only AWS has been rotated; Supabase, Stripe, Twilio, LiveKit, Sentry have not been.
10. **The IR plan describes external notifications but has not been walked through end-to-end.** For example, the OPC's actual online form has fields the current template does not enumerate.
